# Ops Agent 工作流文档

本文档描述 Ops Agent 的核心用户工作流，用于指导产品设计、后端接口、前端交互、桌面端能力和后续开发排期。

## 1. 资产管理工作流

### 目标

用户可以维护需要运维的资产，并为后续终端连接、LLM 对话和任务执行提供基础上下文。

### 适用资产

- Linux 服务器
- 网络设备
- 其他后续扩展的主机或设备类型

### 用户流程

1. 用户进入资产管理页面。
2. 用户创建资产，填写资产基础信息：
   - 名称
   - 类型
   - 地址
   - 端口
   - 用户名
   - 密钥
   - 密码
   - 备注
3. 系统保存资产信息和连接凭据。
4. 用户可以查看资产列表。
5. 用户可以查看单个资产详情。
6. 用户可以编辑资产信息。
7. 用户可以删除不再使用的资产。

### 修改能力

用户编辑资产时，至少需要支持修改：

- 地址
- 端口
- 用户名
- 密钥
- 密码
- 备注

### 系统行为

- 创建资产时校验必填字段。
- 密钥和密码需要安全保存，不应以明文形式暴露在界面和日志中。
- 删除资产前需要二次确认。
- 删除资产后，不保留该资产的历史会话；审计记录需要按保留周期继续保存，便于用户查询历史操作。

### 结果

资产成为后续终端连接、LLM 对话、命令执行和审计记录的基础对象。

## 2. 大模型配置工作流

### 目标

用户可以配置一个或多个大模型提供商，让系统在不同场景下选择合适的 LLM。

### 配置项

每个大模型配置至少包含：

- 名称
- 提供商
- Base URL
- API Key
- Model
- 是否默认启用
- 备注

### 用户流程

1. 用户进入模型配置页面。
2. 用户创建大模型配置。
3. 用户填写 provider、url、apikey、model 等信息。
4. 用户保存配置。
5. 系统校验配置格式。
6. 用户可以查看模型配置列表。
7. 用户可以查看单个模型配置详情。
8. 用户可以修改模型配置。
9. 用户可以删除模型配置。
10. 用户可以选择默认模型配置。

### 增删改查能力

- 新增模型配置。
- 查询模型配置列表。
- 查询模型配置详情。
- 修改模型提供商、url、apikey、model。
- 删除模型配置。

### 系统行为

- API Key 需要脱敏展示。
- API Key 不应写入普通日志。
- 系统只有一个全局默认模型。
- 用户配置多个模型时，可以在对话界面切换当前使用的模型。
- 删除默认模型前需要提示用户先选择新的默认模型。
- 保存配置时可以提供“测试连接”能力。
- 模型配置应支持 Anthropic、OpenAI Compatible，并预留其他 provider 扩展能力。

### 结果

系统可以基于用户配置调用指定 LLM，完成对话、计划生成、命令建议和结果分析。

## 3. 资产连接与上下文对话工作流

### 目标

用户点击资产后，系统连接该资产，并让 LLM 使用该资产上下文进行自然语言对话。

### 用户流程

1. 用户在资产列表中点击一个资产。
2. 系统读取该资产的基础信息和连接凭据。
3. 系统打开该资产的终端会话。
4. 系统获取资产上下文，例如：
   - 资产名称
   - 资产类型
   - 地址
   - 操作系统或设备类型
   - 当前终端连接状态
   - 最近终端输出
   - 历史助手会话
5. 用户在 AI 对话框中输入自然语言问题。
6. LLM 基于资产上下文回答用户。
7. 用户继续追问，系统持续维护当前资产上下文。

### 系统行为

- 每个资产连接应对应一个可追踪的终端会话。
- LLM 请求中需要包含当前资产上下文。
- 终端输出可以被选择并附加到 LLM 对话中。
- 当资产连接失败时，LLM 应知道当前资产不可执行命令，只能提供排查建议。

### 结果

用户可以围绕当前资产进行上下文相关的运维问答，而不是进行孤立的普通聊天。

## 4. LLM 计划、审批、命令执行与反馈工作流

### 目标

用户输入任务后，LLM 自动创建计划，给出建议命令。用户审批后，命令在资产终端执行，执行结果返回给 LLM，直到任务完成。

### 用户流程

1. 用户选择一个资产并建立终端连接。
2. 用户输入自然语言任务，例如“帮我检查这台机器磁盘为什么快满了”。
3. LLM 理解任务目标和当前资产上下文。
4. LLM 创建执行计划。
5. 系统展示计划步骤。
6. LLM 针对当前步骤生成建议命令。
7. 系统展示待执行命令和风险说明。
8. 用户点击审批同意。
9. 系统将命令发送到当前资产终端执行。
10. 系统采集命令输出、退出状态和执行时间。
11. 执行结果返回给 LLM。
12. LLM 分析结果并决定下一步：
    - 继续生成下一条命令
    - 要求用户补充信息
    - 给出结论
    - 标记任务完成
13. 系统持续循环，直到用户任务完成或用户手动停止。

### 审批节点

默认情况下，服务器和网络设备上的所有终端命令都需要用户审批。

审批内容至少包含：

- 命令内容
- 执行资产
- 执行目录或上下文
- 风险等级
- LLM 解释
- 预期输出

### 系统行为

- 未经审批的命令不得执行。
- 命令执行结果必须完整回传给 LLM。
- 每一步计划、审批、命令和结果都需要持久化。
- 用户可以拒绝某条命令，并要求 LLM 重新规划。
- 用户可以中止整个任务。

### 结果

用户保留最终控制权，LLM 负责规划、建议、分析和推进任务。

## 5. 多资产协作与 Sub-Agent 工作流

### 目标

用户在与 LLM 对话过程中，可以让 LLM 连接其他机器或设备协助完成任务。其他资产的终端会话作为 sub-agent 执行子任务，并将进度和结果返回给主 agent。

### 典型场景

- 当前机器需要访问数据库服务器确认状态。
- 应用服务器异常，需要同时查看负载均衡设备配置。
- 网络故障排查需要同时查看交换机、路由器、防火墙状态。
- 主机问题需要跨多台机器对比日志或配置。

### 用户流程

1. 用户在当前资产对话中提出任务。
2. LLM 判断该任务需要其他资产协助。
3. LLM 向用户说明需要连接哪些资产以及原因。
4. 用户选择或确认需要协作的资产。
5. 系统为每个协作资产打开独立终端会话。
6. 主 agent 为每个协作资产创建子任务。
7. 每个 sub-agent 基于自己的资产上下文生成计划和建议命令。
8. sub-agent 生成的每条命令都需要用户点击审批。
9. sub-agent 在对应资产终端执行命令。
10. sub-agent 将执行结果、当前进度和结论返回主 agent。
11. 主 agent 汇总多个资产结果。
12. 主 agent 继续推进整体任务，直到完成。

### 系统行为

- 主 agent 负责整体目标、任务拆分和最终总结。
- sub-agent 负责单个资产上的局部任务。
- 每个 sub-agent 必须绑定明确的资产和终端会话。
- 不同资产的命令、审批和输出需要隔离记录。
- 主 agent 只能基于 sub-agent 返回的结果推进判断，不能混淆资产上下文。

### 结果

系统支持跨资产协同运维，能够处理需要多台机器或多类设备共同参与的复杂任务。

## 6. 自动审批规则工作流

### 目标

用户可以在当前大模型对话会话中配置哪些命令不需要人工审批。符合当前会话规则的低风险命令可以由 LLM 自动执行，提高运维效率。

### 用户流程

1. 用户在当前对话会话中进入自动审批配置。
2. 用户创建仅对当前会话生效的自动审批规则。
3. 用户配置规则条件，例如：
   - 资产类型
   - 资产标签
   - 命令名称
   - 命令参数
   - 风险等级
   - 只读命令
   - 最大执行时长
4. 用户保存规则。
5. LLM 在当前会话中生成建议命令后，系统先匹配当前会话的自动审批规则。
6. 如果命中规则，系统跳过本次人工审批并自动执行。
7. 如果未命中规则，系统进入人工审批。
8. 自动执行结果仍然返回给 LLM。
9. 用户可以在当前会话中随时禁用、修改或删除自动审批规则。

### 推荐默认策略

默认只允许低风险只读命令进入自动审批，例如：

- 查看系统信息
- 查看磁盘使用率
- 查看内存和 CPU 状态
- 查看进程列表
- 查看网络连接状态
- 查看网络设备运行状态

### 禁止默认自动审批的命令

以下类型命令默认不应自动审批：

- 删除文件
- 修改配置
- 重启服务
- 关闭进程
- 修改网络配置
- 写入磁盘
- 修改用户或权限
- 执行未知脚本
- 批量变更设备配置

### 系统行为

- 自动审批规则的作用范围限定为当前大模型对话会话。
- 自动审批执行也必须进入审计日志。
- 自动审批命令需要可追溯到具体规则。
- 用户应能查看某条命令为什么被自动审批。
- 高风险命令不能仅凭 LLM 判断自动执行。

### 结果

低风险、重复性、只读类命令可以自动执行，高风险操作仍由用户控制。

## 7. 网络设备操作工作流

### 目标

用户可以通过 Ops Agent 操作多厂商网络设备，包括思科、华为、Juniper、华三等设备。

### 支持设备类型

首批目标网络设备包括：

- Cisco
- Huawei
- Juniper
- H3C

后续可扩展：

- Arista
- Fortinet
- Palo Alto
- MikroTik
- 其他支持 SSH、NETCONF、RESTCONF 或厂商 API 的设备

### 用户流程

1. 用户创建网络设备资产。
2. 用户选择设备厂商和设备类型。
3. 用户填写管理地址、端口、用户名、密码或密钥。
4. 用户点击设备资产。
5. 系统建立网络设备终端或 API 会话。
6. 系统识别设备厂商和命令模式。
7. 用户输入自然语言任务，例如：
   - “检查这个交换机端口是否有错误包”
   - “查看 BGP 邻居状态”
   - “帮我确认这台防火墙是否有丢包”
8. LLM 根据设备厂商生成对应命令。
9. 用户审批命令。
10. 系统在网络设备上执行命令。
11. 设备输出返回给 LLM。
12. LLM 分析结果并给出下一步建议或最终结论。

### 设备厂商差异

系统需要处理不同厂商的差异，包括：

- 命令语法不同
- 分页输出不同
- 配置模式不同
- show/display 命令差异
- 接口命名差异
- 返回格式差异
- 权限模式差异

### 系统行为

- 每个厂商应有独立的命令模板和解析策略。
- LLM 生成命令前需要知道设备厂商。
- 网络设备配置变更类命令默认需要人工审批。
- 对网络设备执行批量变更前需要更严格的确认。
- 对只读巡检命令可以结合自动审批规则。

### 结果

用户可以通过自然语言对多厂商网络设备进行巡检、排障和受控变更。

## 8. 端到端主流程

下面是完整串联后的主流程：

1. 用户配置 LLM provider、url、apikey、model。
2. 用户创建资产并保存连接凭据。
3. 用户点击资产，系统建立终端会话。
4. 系统将资产上下文传给 LLM。
5. 用户输入自然语言运维任务。
6. LLM 创建执行计划。
7. LLM 生成建议命令。
8. 系统判断命令是否命中当前会话的自动审批规则。
9. 如果命中当前会话自动审批规则，系统直接执行命令。
10. 如果未命中规则，系统等待用户审批。
11. 命令在资产终端执行。
12. 执行结果返回给 LLM。
13. LLM 分析结果并继续下一步。
14. 如果需要其他资产协作，主 agent 创建 sub-agent 工作流。
15. sub-agent 在其他资产上执行子任务并返回结果。
16. 主 agent 汇总所有结果。
17. LLM 给出最终结论、风险说明和后续建议。
18. 系统保存完整会话、计划、审批、命令、输出和总结。

## 9. 后续开发拆分建议

### 第一阶段：补齐基础 CRUD

- 资产 CRUD。
- 凭据安全保存。
- 模型配置 CRUD。
- 默认模型选择。
- 基础连接测试。

### 第二阶段：单资产 Agent 闭环

- 点击资产建立终端连接。
- 资产上下文注入 LLM。
- LLM 生成计划。
- 命令审批。
- 命令执行。
- 输出返回 LLM。
- 任务完成总结。

### 第三阶段：自动审批

- 命令风险分级。
- 自动审批规则 CRUD。
- 规则匹配引擎。
- 自动审批审计日志。
- 自动审批可解释展示。

### 第四阶段：多资产 Sub-Agent

- 多终端会话管理。
- 主 agent / sub-agent 任务模型。
- 子任务分派。
- 子任务进度回传。
- 多资产结果汇总。

### 第五阶段：网络设备增强

- Cisco 支持。
- Huawei 支持。
- Juniper 支持。
- H3C 支持。
- 厂商命令模板。
- 网络设备输出解析。
- 网络设备变更审批策略。

## 10. 已确认产品决策

以下决策已确认，后续详细设计和实现应以此为准：

1. 资产删除后，不保留该资产的历史会话；审计记录需要保留一段时间，以便用户查询历史操作。
2. 密码不需要支持轮换历史；用户可以直接修改密码。密钥需要由用户管理，并绑定到具体服务器资产。
3. 模型配置采用全局唯一默认模型；如果用户配置了多个模型，可以在对话界面手动切换当前模型。
4. 自动审批规则按照当前大模型对话会话生效，不作为全局规则、团队规则或资产永久规则。
5. sub-agent 执行的每条命令都需要用户点击审批，审批通过后才能执行。
6. 网络设备和服务器设备都允许执行命令，但所有命令都需要遵循审批策略。
7. Web 前端和桌面端都需要完整支持本文档中的核心工作流。
